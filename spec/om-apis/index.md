<h2 id="use-strings-not-numeric-constants">Use Strings, not Numeric Constants</h2>
<p>
When an <abbr title="Application Programming Interface">API</abbr> needs to take or return an enumerated value, many older APIs follow the C convention and use numeric constants, stored on some interface object.  Modern <abbr title="Application Programming Interface">API</abbr> practice is instead to use strings directly.  See, for example, the <code>responseType</code> property used in XHR.
</p>

<p>
An anti-example can be found in CSSTransformValue, which returns a list of transforms that expose an <code>operationType</code> property.  This is currently specified to contain an integer, with the intent that authors testing the value will use the constants like <code>CSSTransformValue.<abbr title="Cascading Style Sheets">CSS</abbr>_TRANSLATE3D</code>.
</p>

<p>
There are multiple problems with this:
</p>
<ol>
<li class="level1">it is much longer to type than a string like “translate3d”</li>
<li class="level1">the name of the constant is different than the name of the function it corresponds to, while a string can be the same.</li>
<li class="level1">simply printing or storing the value reveals an opaque integer like <code>13</code>, which can&#039;t be translated back into a transform type without testing against every constant</li>
<li class="level1">long experience shows that, given the choice of using a named constant or an integer, authors will end up using the integer form instead, often due to cargo-cult “speed improvements”</li>
</ol>

<p>
Instead of code like:
</p>
<pre class="code">  if (list.operationType == CSSTransformValue.CSS_TRANSLATE3D)</pre>

<p>
You can instead have code like:
</p>
<pre class="code">  if (list.operationType == &quot;translate3d&quot;)</pre>
